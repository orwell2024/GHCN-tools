def fetch_file_content(url, is_text=True):
    logging.info(f"Fetching data from: {url}")
    try: response = requests.get(url, timeout=60, verify=True); response.raise_for_status(); logging.info("Data fetched successfully.")
    except requests.exceptions.SSLError as e:
        logging.error(f"SSL Error fetching {url}: {e}. Trying verify=False.");
        try: response = requests.get(url, timeout=60, verify=False); response.raise_for_status(); logging.info("Fetched (no SSL verify).")
        except requests.exceptions.RequestException as e_retry: logging.error(f"Retry fetch failed {url}: {e_retry}"); return None
    except requests.exceptions.RequestException as e: logging.error(f"Error fetching {url}: {e}"); return None
    if is_text:
        detected_encoding = response.apparent_encoding
        try: content = response.content.decode(detected_encoding if detected_encoding else 'utf-8')
        except (UnicodeDecodeError, TypeError):
            try: content = response.content.decode('latin1')
            except UnicodeDecodeError: content = response.content.decode('cp1252', errors='ignore')
        return content
    else: return response.content
def haversine(lon1, lat1, lon2, lat2):
    if any(pd.isna(x) for x in [lon1, lat1, lon2, lat2]): return float('inf')
    lon1, lat1, lon2, lat2 = map(radians, [lon1, lat1, lon2, lat2]); dlon = lon2 - lon1; dlat = lat2 - lat1; a = sin(dlat/2)**2 + cos(lat1) * cos(lat2) * sin(dlon/2)**2
    if a < 0: a = 0;
    if a > 1: a = 1;
    c = 2 * asin(sqrt(a)); r = 6371; return c * r
def latlon_to_cartesian(lat, lon):
    lat_rad = np.radians(lat); lon_rad = np.radians(lon); x = np.cos(lat_rad) * np.cos(lon_rad); y = np.cos(lat_rad) * np.sin(lon_rad); z = np.sin(lat_rad); return x, y, z

def haversine(lon1, lat1, lon2, lat2):
    if any(pd.isna(x) for x in [lon1, lat1, lon2, lat2]): return float('inf')
    lon1, lat1, lon2, lat2 = map(radians, [lon1, lat1, lon2, lat2]); dlon = lon2 - lon1; dlat = lat2 - lat1; a = sin(dlat/2)**2 + cos(lat1) * cos(lat2) * sin(dlon/2)**2
    if a < 0: a = 0;
    if a > 1: a = 1;
    c = 2 * asin(sqrt(a)); r = 6371; return c * r
def latlon_to_cartesian(lat, lon):

This notebook performed a multi-stage matching process to link stations from a predefined list of "very rural" Berkeley Earth (BE) sites to corresponding records in the official GHCN Daily Inventory and a specific GHCNv4 Meta file containing Built-Up/Imperviousness data.

Workflow Recap:

  1. Data Loading:
    • Loaded a list of target "Rural" BE station IDs from Climate Audit.
    • Loaded the full Berkeley Earth station details file.
    • Loaded the official GHCN Daily station inventory (ghcnd-inventory.txt) from NOAA/NCEI.
    • Loaded a specific GHCNv4 metadata file containing pre-calculated surface characteristics (GHCNv4_stations_with_BI_BU_orwell2022.csv).
  2. BE Filtering: The full BE station list was filtered to include only the target "Rural" stations.
  3. GHCND Inventory Filtering: The GHCN Daily Inventory was filtered to keep only TMAX records (as a representative record for each station's location and date range) and deduplicated by station ID to create a unique list for spatial matching.
  4. Coordinate Cleaning: Latitude and Longitude values in all relevant dataframes were converted to numeric types, and entries with invalid coordinates were removed.
  5. Spatial Matching (BE -> GHCND Inventory):
    • A KD-Tree spatial index was built on the coordinates of the unique GHCND Inventory (TMAX) stations for efficient searching.
    • Each Rural BE station was then queried against this tree to find its single nearest neighbor in the GHCND Inventory.
    • The Haversine (great-circle) distance was calculated between the BE station and its closest GHCND Inventory neighbor.
    • If this distance was less than or equal to the threshold (MAX_DISTANCE_KM = 2.0 km), the (BE_ID, GHCND_INV_ID, Distance) pair was considered a potential match.
  6. ID Linking (GHCND Inventory -> GHCN Meta):
    • The GHCND_INV_ID from the successfully matched pairs was used to look up corresponding records in the loaded GHCNv4 Meta file based on an exact ID match.
  7. Data Merging: The initial BE-GHCND pairs were progressively merged with details from the GHCND Inventory (TMAX record), the GHCN Meta file, and the full details of the original Rural BE station file to create a final comprehensive dataset (df_final_triplets).

Performance Optimization with KD-Tree:

Initially, calculating the distance from each Rural BE station to every potential match in the large GHCN Daily Inventory file (~130,000 unique stations after filtering) presented a significant performance bottleneck. A brute-force approach would involve billions of distance calculations, leading to impractically long execution times (estimated in hours or days).

To address this, the following key prompt was provided:

User Prompt Snippet: "We may need treecode to speed up"

This insight correctly identified the need for a spatial indexing method. In response, the implementation was modified to use a KD-Tree, a data structure optimized for nearest neighbor searches in multi-dimensional space:

Gemini Response Snippet (Conceptual): "Okay, you're right. Calculating all pairwise distances can be very slow... Using a spatial index like a KD-Tree is an excellent way to significantly speed up the process... KD-Tree Approach:

  1. We'll build a KD-Tree using the coordinates of the... [GHCND Inventory stations].
  2. For each [Rural BE station], we'll query the KD-Tree to find the index of the single closest [GHCND Inv] station...
  3. We then calculate the actual great-circle (Haversine) distance only between the [BE] station and that one closest [GHCND Inv] station...
  4. We check if this distance meets the... threshold."

(The subsequent code block implemented this logic using scipy.spatial.KDTree)

This optimization dramatically reduced the computational complexity. Instead of comparing every BE station to every GHCND station (O(N*M)), the KD-Tree allows finding the nearest neighbor for each BE station in approximately logarithmic time relative to the size of the GHCND dataset (closer to O(M log N)). As demonstrated by the performance comparison script (if run), this reduces the matching time from potentially hours/days to minutes, making the analysis feasible. This highlights the crucial role of appropriate algorithms when dealing with large spatial datasets.
